This one has an architectural decision embedded in it worth surfacing before I write the plan.

The current solution stores Spotify tokens in Firestore under `users/{userId}/spotifyTokens`. There are two approaches to making this safe:

**Option A — Strict Firestore security rules + keep tokens in Firestore.** Lock down the rules so only the authenticated user can read their own tokens. This is the simplest change to what's already designed. The risk is that security rules are easy to misconfigure during development and easy to accidentally loosen later.

**Option B — Move tokens out of Firestore entirely.** Store them only in the Cloud Function environment (as HttpOnly session cookies or in Secret Manager). The frontend never touches tokens directly — it just holds a session cookie, and the Cloud Function handles all Spotify API calls. This is architecturally more secure: even if Firestore rules are misconfigured, there are no tokens to steal. The tradeoff is that it pushes more logic into Cloud Functions, since the frontend can no longer call the Spotify API directly.

The current architecture has the **frontend calling Spotify's API directly** (fetching albums, etc.), with Cloud Functions only handling OAuth. If we move tokens server-side, those Spotify API calls would need to go through Cloud Functions too — a meaningful architectural shift.

- Keep the current architecture, lock it down with strict Firestore rules (simpler, some residual risk)

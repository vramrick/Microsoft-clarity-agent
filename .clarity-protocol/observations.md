# Observations

## Interaction Map Between Failures

FM01 (disengagement) is upstream of FM02-04: when users disengage, AI quality goes unchecked, alignment fails silently, analysis stays shallow. Managing FM01 has the highest leverage.

FM02 (AI quality) feeds back into FM01: if AI output is shallow, disengagement is rational. The two form a reinforcing loop that the early-value and self-critique mechanisms aim to break.

FM05 (organizational misuse) is partially addressed by the same mechanisms as FM01-04 but has irreducible organizational components.

FM06 (drift) and FM07 (operational) are independent — engineering/infrastructure problems with standard solutions.

**Key conflict:** Challenge intensity (FM01 wants less, FM02/03 want more). Resolution: adaptive calibration based on user expertise signals, not a static setting.

**Key conflict:** Conciseness vs. completeness (FM01/03 want shorter, FM04/05 want thorough). Resolution: summary layers in the protocol format — critical content first, detail on demand.

## Induced Failures

One potential induced failure identified during management planning:

- **Self-critique over-correction** (from FM02 management plan): Invisible self-critique could cause the agent to become over-hedged, slow, or afraid to commit to positions. Mitigated through critique guide calibration — the disposition should be "notice when something feels off" not "question everything." Filed in suggestion box for Layer 2 design phase.

## Ungrouped Failures

**Scope creep into project management.** Users who experience the clarity process naturally want the tool to help with next steps — task tracking, sprint planning, work assignment. They start overloading protocol documents with implementation details. This is a boundary management issue rather than a failure of the core process. Low severity; best handled through clear communication about what the tool is and isn't.

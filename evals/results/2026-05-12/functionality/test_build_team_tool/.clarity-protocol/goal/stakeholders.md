# Stakeholders

## Engineers (direct, aligned — primary)

7 engineers (3 senior, 3 mid, 1 junior), mostly remote with some overlapping hours. Primary users. They post updates and read each other's. Their core need is coordination: knowing who's in what area so they don't duplicate work or make conflicting assumptions. They'll use the tool voluntarily under a team norm (like standups), but friction is the enemy — if updates are too burdensome, compliance will drop.

## Engineering Manager (direct, aligned — secondary)

Wants a quick, reliable view of team state for leadership conversations and personal awareness. This is a *side effect* of the coordination tool, not its primary purpose. Explicitly does not want this to feel like a surveillance mechanism — the EM's visibility need should be served by the same data engineers post for each other.

## VP / Leadership (indirect, aligned)

Occasionally asks "who's working on what." Not active users; they receive summarized information from the EM. No direct interaction with the tool.

## Adversarial / misuse

Low risk given internal-only scope. One design tension: if updates feel like they're being watched or evaluated, engineers may game them (post vague or flattering updates) or stop updating entirely. This is the key adversarial dynamic to guard against — the tool undermining its own data quality through perceived surveillance pressure.
